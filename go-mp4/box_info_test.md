# 🧪 Тести `WriteBoxInfo` / `ReadBoxInfo`: Повний розбір

Це **юніт-тести** для функцій серіалізації/десеріалізації заголовків MP4-боксів. Вони перевіряють, що бібліотека коректно працює з усіма крайніми випадками формату ISO/IEC 14496-12.

---

## 📋 Структура тесту

```go
func TestWriteBoxInfo(t *testing.T) {
    // 1. Оголошуємо структуру тест-кейсу
    type testCase struct {
        name          string      // Назва тесту (для логу)
        pre           []byte      // Дані, записані ДО боксу (для тесту offset)
        bi            *BoxInfo    // Вхідний BoxInfo для запису
        hasError      bool        // Чи очікуємо помилку?
        expectedBI    *BoxInfo    // Який BoxInfo очікуємо на виході?
        expectedBytes []byte      // Які байти мають бути записані?
    }

    // 2. Масив тест-кейсів
    testCases := []testCase{ {...}, {...}, ... }

    // 3. Запуск кожного кейсу
    for _, c := range testCases {
        t.Run(c.name, func(t *testing.T) {
            // Тестова логіка...
        })
    }
}
```

> 🎯 **Pattern**: "Table-driven tests" — стандартний підхід у Go для тестування функцій з багатьма вхідними варіантами.

## 🔍 Детальний розбір тест-кейсів для `WriteBoxInfo`

### ✅ Кейс 1: "small-size" — малий заголовок (8 байт)

```go
{
    name: "small-size",
    bi: &BoxInfo{
        Size:       0x12345,              // 74 565 байт (< 4 ГБ)
        HeaderSize: 8,                     // очікуємо малий заголовок
        Type:       StrToBoxType("test"), // тип боксу: "test"
    },
    expectedBytes: []byte{
        0x00, 0x01, 0x23, 0x45,  // size = 0x00012345 (big-endian!)
        't', 'e', 's', 't',      // type = "test"
    },
}
```

**Що перевіряємо:**
```
📦 Заголовок боксу (< 4 ГБ):
┌─────────────────┐
│ 00 01 23 45     │ ← 4 байти: розмір (uint32, big-endian)
├─────────────────┤
│ 74 65 73 74     │ ← 4 байти: тип ("test" в ASCII)
└─────────────────┘
     ↓
     8 байт заголовка ✅
```

---

### ✅ Кейс 2: "large-size" — великий заголовок (16 байт)

```go
{
    name: "large-size",
    bi: &BoxInfo{
        Size:       0x123456789abc,  // 20 014 999 342 524 байт (> 4 ГБ!)
        HeaderSize: 8,                // вхідне значення (ігнорується)
        Type:       StrToBoxType("test"),
    },
    expectedBI: &BoxInfo{
        Size:       0x123456789abc + 8,  // +8 байт компенсації!
        HeaderSize: 16,                   // автоматично змінено на 16!
        Type:       StrToBoxType("test"),
    },
    expectedBytes: []byte{
        // Великий заголовок (16 байт):
        0x00, 0x00, 0x00, 0x01,  // size=1 → маркер "читай 64-бітний розмір"
        't', 'e', 's', 't',      // type = "test"
        0x00, 0x00, 0x12, 0x34,  // 64-бітний розмір (продовження):
        0x56, 0x78, 0x9a, 0xbc,  // 0x0000123456789abc
    },
}
```

**🎯 Ключова логіка:**
```
Якщо Size > 0xFFFFFFFF (4 ГБ):
1. Записати size=1 у перші 4 байти (маркер)
2. Записати тип боксу (4 байти)
3. Записати реальний 64-бітний розмір (8 байт)
4. Повернути оновлений BoxInfo з HeaderSize=16
```

> ⚠️ **Важливо**: `expectedBI.Size = original + 8` — тому що при переході з 8→16 байт заголовка, загальний розмір боксу збільшується на 8 байт!

---

### ✅ Кейс 3: "extend to eof" — бокс до кінця файлу

```go
{
    name: "extend to eof",
    bi: &BoxInfo{
        Size:        0x123,           // тимчасове значення
        HeaderSize:  8,
        Type:        StrToBoxType("test"),
        ExtendToEOF: true,            // 🔹 КЛЮЧОВЕ ПРАПОР
    },
    expectedBytes: []byte{
        0x00, 0x00, 0x00, 0x00,  // size=0 → "тягнеться до EOF"
        't', 'e', 's', 't',      // type = "test"
    },
}
```

**Коли це використовується:**
- Останній `mdat` у фрагментованому MP4 (fMP4)
- Стрімінг, де розмір даних невідомий заздалегідь
- Файли, що записуються "на льоту"

```
📡 HLS-сегмент записується в реальному часі:
1. Записуємо заголовок mdat з size=0, ExtendToEOF=true
2. Стрімимо відео/аудіо байти
3. Коли стрімінг закінчується — файл автоматично "закривається"
```

### ✅ Кейс 4: "with offset" — перевірка зміщення

```go
{
    name: "with offset",
    pre:  []byte{0x00, 0x00, 0x00},  // 3 байти "сміття" перед боксом
    bi: &BoxInfo{
        Size:       0x12345,
        HeaderSize: 8,
        Type:       StrToBoxType("test"),
    },
    expectedBI: &BoxInfo{
        Offset:     3,  // 🔹 Offset має дорівнювати довжині pre!
        Size:       0x12345,
        HeaderSize: 8,
        Type:       StrToBoxType("test"),
    },
    expectedBytes: []byte{
        0x00, 0x00, 0x00,        // pre (3 байти)
        0x00, 0x01, 0x23, 0x45,  // заголовок боксу
        't', 'e', 's', 't',
    },
}
```

**Навіщо це тестувати?**
- У реальних файлах бокси не завжди починаються з нульового байту
- При модифікації існуючих файлів треба знати точне `Offset`
- Для коректного `Seek()` при парсингу

### ✅ Кейс 4: "extend to eof" — імітація кінця файлу

```go
{
    name: "extend to eof",
    buf: []byte{
        0x00, 0x00, 0x00, 0x00,  // size=0 → ExtendToEOF
        't', 'e', 's', 't',      // type
        // Далі — "дані" боксу (у тесті — просто нулі)
        0x00, 0x00, 0x00, 0x00,
        0x00, 0x00, 0x00, 0x00,
        0x00, 0x00, 0x00, 0x00,
    },
    expected: &BoxInfo{
        Size:        20,           // ✅ 8 (header) + 12 (дані) = 20
        HeaderSize:  8,
        Type:        StrToBoxType("test"),
        ExtendToEOF: true,         // ✅ Прапорець встановлено!
    },
}
```

**Логіка `ReadBoxInfo` для `size=0`:**
```go
if bi.Size == 0 {
    // 1. Шукаємо кінець файлу
    eof, _ := r.Seek(0, io.SeekEnd)
    
    // 2. Обчислюємо реальний розмір: EOF - Offset
    bi.Size = uint64(eof) - bi.Offset
    
    // 3. Встановлюємо прапорець
    bi.ExtendToEOF = true
    
    // 4. Позиціонуємось на початок даних
    bi.SeekToPayload(r)
}
```

## 🛠️ Інструменти тестування

### 🔹 `writerseeker.WriterSeeker`

```go
import "github.com/orcaman/writerseeker"

w := &writerseeker.WriterSeeker{}
```

**Що це?** Моковий об'єкт, що реалізує **одночасно** `io.Writer` + `io.Seeker` + `io.Reader`.

| Метод | Призначення |
|-------|-------------|
| `w.Write(data)` | Записує байти у внутрішній буфер |
| `w.Seek(offset, whence)` | Зміщує "курсор" у буфері |
| `w.Reader()` | Повертає `io.Reader` для читання записаного |

> ✅ **Перевага**: Не треба створювати тимчасові файли на диску — все в пам'яті, швидко та ізольовано.

---

### 🔹 `require` vs `assert` (з `stretchr/testify`)

```go
// require — зупиняє тест при помилці (fatal)
require.NoError(t, err)  // якщо err != nil → тест завершується

// assert — продовжує тест, лише логує помилку
assert.Equal(t, expected, actual)  // якщо не співпадає → тест "падає" в кінці
```

**Патерн використання:**
```go
// 1. Спочатку require для критичних перевірок
bi, err := WriteBoxInfo(w, input)
require.NoError(t, err)  // якщо помилка — далі немає сенсу

// 2. Потім assert для валідації результату
assert.Equal(t, expectedBI, bi)           // перевірка структури
assert.Equal(t, expectedBytes, output)    // перевірка байтів
```

## 🎯 Чому ці тести критичні для вашого HLS-процесора?

### 🔹 Сценарій 1: Коректна синхронізація таймстемпів

```
📡 Ви читаєте fMP4-сегмент з камери:
1. ReadBoxInfo правильно визначає HeaderSize (8 або 16)
2. SeekToPayload() точно позиціонує на дані `trun`
3. Ви читаєте PTS/DTS без зсуву на 1 байт ✅

❌ Без тестів: помилка в 1 байт → десинхронізація аудіо/відео
```

### 🔹 Сценарій 2: Запис нових сегментів для HLS

```
📝 Ви генеруєте новий сегмент після обробки:
1. WriteBoxInfo автоматично обирає 8/16-байтний заголовок
2. Коригує Size з урахуванням реального заголовка
3. Плеєр (VLC, hls.js) коректно читає ваш файл ✅

❌ Без тестів: неправильний size → плеєр "зависає" на сегменті
```

### 🔹 Сценарій 3: Обробка "обрваних" сегментів

```
⚠️ Мережева помилка: останній mdat обірвався:
1. ReadBoxInfo бачить size=0 → встановлює ExtendToEOF=true
2. Ви коректно обробляєте "неповний" сегмент
3. Логуєте попередження, але не крашите сервер ✅

❌ Без тестів: panic при Seek(EOF) → сервер падає
```

## ⚠️ Поширені помилки при написанні тестів

| Помилка | Наслідок | Як уникнути |
|---------|----------|-------------|
| Неправильний byte order | Тест проходить, але реальний файл не читається | Завжди використовуйте `binary.BigEndian` для MP4 |
| Забули `+8` для large-size | `expectedBI.Size` не співпадає | Пам'ятайте: 16-байтний заголовок = +8 байт до розміру |
| Не тестуєте `ExtendToEOF` | Помилки при обробці останнього сегмента | Додайте кейс з `size=0` |
| Ігноруєте `Offset` | Проблеми при модифікації існуючих файлів | Тестуйте з `pre`-даними |

```

        log.Printf("⚠️ HeaderSize changed: %d → %d", 
            bi.HeaderSize, updated.HeaderSize)
    }

[ ] Для обробки мережевих помилок: перевіряйте ExtendToEOF:
    if bi.ExtendToEOF {
        log.Printf("🔚 Last box %s extends to EOF", bi.Type)
        // Обробити "неповний" сегмент
    }

[ ] Додайте свій тест-кейс для специфічних боксів вашого стріму:
    • Бокси з китайськими метаданими
    • Специфічні кодеки (AV1, HEVC)
    • Великі сегменти (>4 ГБ для архівування)
```
