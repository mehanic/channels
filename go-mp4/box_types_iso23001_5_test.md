# 🧪 Тест `TestBoxTypesISO23001_5`: PCM Configuration Box (`pcmC`)

Це **юніт-тест** для бібліотеки `go-mp4`, який перевіряє коректну роботу **серіалізації/десеріалізації** боксу `pcmC` (PCM Configuration) згідно зі стандартом **ISO/IEC 23001-5** (Common Encryption та аудіо-конфігурації).

---

## 🎯 Коротка відповідь

> **Цей тест гарантує, що параметри PCM-аудіо (ціле/плаваюче, розмір семпла) коректно перетворюються між структурою Go та 6-байтовим бінарним форматом** — критично для відтворення некомпресованого аудіо у плеєрах.

## 🔍 Детальний розбір тест-кейсу: "pcmC"

### ✅ Вхідні дані (`src`) — конфігурація PCM

```go
src: &PcmC{
    FormatFlags:   1,      // 🔹 Float PCM, Little-endian
    PCMSampleSize: 32,     // 🔹 32 біти на семпл
},
```

**📊 Що це означає:**

| Поле | Значення | Бітова інтерпретація |
|------|----------|---------------------|
| `FormatFlags = 1` | `0x01` = `0000 0001` | Біт 0 = 1 → **Float PCM** (не integer!) |
| `PCMSampleSize = 32` | `0x20` = 32 десяткове | **32-бітні семпли** (плаваюча кома) |

> 🎯 **Важливо**: `FormatFlags = 1` означає **Float PCM**, а не Integer! Це ключова відмінність для декодера.

---

### ✅ Очікувані байти (`bin`) — 6 байт заголовка + даних

```go
bin: []byte{
    0x0,                // 🔹 FullBox: version = 0
    0x00, 0x00, 0x00,   // 🔹 FullBox: flags = 0x000000
    0x1,                // 🔹 FormatFlags = 1 (Float PCM)
    0x20,               // 🔹 PCMSampleSize = 32 (0x20 = 32 десяткове)
},
```

**🔢 Детальна розбивка:**

```
📦 pcmC бокс (всього 6 байт):
┌─────────────────────────────────┐
│ Байт 0: Version = 0             │
├─────────────────────────────────┤
│ Байти 1-3: Flags = 0x000000     │
├─────────────────────────────────┤
│ Байт 4: FormatFlags = 0x01      │
│         [0000 0001]             │
│         ↑                       │
│         Біт 0 = 1 → Float PCM   │
├─────────────────────────────────┤
│ Байт 5: PCMSampleSize = 0x20    │
│         [0010 0000] = 32        │
└─────────────────────────────────┘
```

> ⚠️ **Зверніть увагу**: У вашому коді визначення структури `PcmC` обидва поля мають `mp4:"1,size=8"` — це одруківка! Правильно має бути:
> ```go
> FormatFlags   uint8 `mp4:"1,size=8"`   // поле #1
> PCMSampleSize uint8 `mp4:"2,size=8"`   // поле #2 ← має бути "2", не "1"!
> ```

## 🔄 Чотири операції, що тестуються

### 🔹 1. `Marshal` — серіалізація (структура → байти)

```go
buf := bytes.NewBuffer(nil)
n, err := Marshal(buf, tc.src, tc.ctx)
require.NoError(t, err)
assert.Equal(t, uint64(len(tc.bin)), n)  // має бути рівно 6 байт!
assert.Equal(t, tc.bin, buf.Bytes())      // байт в байт: [00 00 00 00 01 20]
```

**Що перевіряємо:**
```
📦 Вхід: PcmC{FormatFlags:1, PCMSampleSize:32}
              │
              ▼
         Marshal() + FullBox заголовок
              │
              ▼
📤 Вихід: []byte{0x0, 0x00,0x00,0x00, 0x1, 0x20}
              │
              ▼
✅ Порівнюємо з еталонним масивом — кожен байт на своєму місці
```

---

### 🔹 2. `Unmarshal` — десеріалізація (байти → структура)

```go
r := bytes.NewReader(tc.bin)
n, err = Unmarshal(r, uint64(len(tc.bin)), tc.dst, tc.ctx)
require.NoError(t, err)
assert.Equal(t, uint64(buf.Len()), n)  // прочитано рівно 6 байт
assert.Equal(t, tc.src, tc.dst)         // 🔁 round-trip: структура відновлена точно!
```

**🔁 Round-trip тест — найважливіша перевірка:**
```
Структура → 6 байт → Структура'

Якщо Структура == Структура' → ✅ серіалізація ідемпотентна
Якщо ні → ❌ втрата даних → декодер отримає хибні параметри → звук "зламається"
```

---

### 🔹 3. `UnmarshalAny` — динамічний парсинг за типом

```go
dst, n, err := UnmarshalAny(
    bytes.NewReader(tc.bin), 
    tc.src.GetType(),        // BoxTypePcmC() = "pcmC"
    uint64(len(tc.bin)), 
    tc.ctx,
)
require.NoError(t, err)
assert.Equal(t, tc.src, dst)
```

**Навіщо `UnmarshalAny`?**
```
🔍 Сценарій: Ви читаєте файл і не знаєте наперед тип боксу

1. ReadBoxInfo() → тип="pcmC", size=6
2. UnmarshalAny(r, "pcmC", 6, ctx) 
3. Бібліотека сама:
   • Знаходить зареєстровану структуру для "pcmC" (PcmC{})
   • Парсить FullBox заголовок (4 байти)
   • Читає FormatFlags (1 байт)
   • Читає PCMSampleSize (1 байт)
   • Повертає готовий об'єкт

✅ Ви не пишете 50+ `case "pcmC":` у своєму коді!
```

---

### 🔹 4. `Stringify` — людський формат для дебагу

```go
str, err := Stringify(tc.src, tc.ctx)
require.NoError(t, err)
assert.Equal(t, tc.str, str)
```

**Приклад використання в логах вашого процесора:**
```go
// ❌ Погано: незрозумілий дамп байтів
log.Printf("pcmC: % x", pcmCBytes)  // [00 00 00 00 01 20]

// ✅ Добре: зрозумілі параметри
log.Printf("🔊 %s", Stringify(pcmC, ctx))
// 🔊 Version=0 Flags=0x000000 FormatFlags=0x1 PCMSampleSize=0x20

// 🔥 Ще краще: людино-читабельна інтерпретація
if pcmC.FormatFlags & 0x08 != 0 {
    log.Printf("🎵 Float PCM detected (%d-bit)", pcmC.PCMSampleSize)
} else {
    log.Printf("🎵 Integer PCM detected (%d-bit)", pcmC.PCMSampleSize)
}
```

```go
// Після Unmarshal: перевіряємо, що курсор в кінці прочитаних даних
s, err := r.Seek(0, io.SeekCurrent)
```

```
✅ Якщо курсор точно в кінці "pcmC" → "mdat" читається коректно
```

## 🎯 Чому цей тест критичний для вашого HLS-процесора?

### 🔹 Сценарій 1: Коректне відтворення Float PCM

```
📡 Камера надсилає fMP4 з 32-біт float PCM:
1. Unmarshal читає pcmC: FormatFlags=1, PCMSampleSize=32
2. Ви передаєте ці параметри у аудіо-декодер
3. Декодер ініціалізується для 32-біт float (не integer!)
4. Клієнт чує чистий звук без артефактів

❌ Без тесту: помилка в бітовій інтерпретації → FormatFlags=1 читається як 0
   → декодер думає, що це integer PCM → інтерпретує біти неправильно → шум/артефакти
```

### 🔹 Сценарій 2: Валідація сумісності з плеєрами

```
🔍 Ви перевіряєте, чи підтримає клієнт цей потік:
1. Читаєте PCMSampleSize = 32
2. Якщо FormatFlags & 0x08 != 0 → Float PCM
3. Перевіряєте: чи підтримує клієнт 32-біт float?
   • Safari/Web Audio API → так ✅
   • Старий браузер → ні ❌ → конвертуєте в 16-біт integer

✅ Тест гарантує, що бітові прапорці читаються коректно
❌ Без тесту: 32-біт float читається як 16-біт → десинхронізація або краш
```

### 🔹 Сценарій 3: Генерація нових fMP4 з правильними параметрами

```
📝 Ви змінюєте якість аудіо на льоту:
1. Прочитали pcmC з вхідного потоку: 32-біт float
2. Змінили на 16-біт integer для економії бітрейту:
   • FormatFlags: 1 → 0 (float → integer)
   • PCMSampleSize: 32 → 16
3. Записали оновлений pcmC у новий сегмент
4. Клієнт отримує легший потік, але звук залишається чітким

✅ Тест гарантує ідемпотентність: зміни не ламають формат
❌ Без тесту: пошкоджений заголовок → декодер падає → немає звуку
```

## ⚠️ Поширені помилки та як їх уникнути

| Помилка | Симптоми | Рішення |
|---------|----------|---------|
| Одруківка в тегах `mp4:"..."` | Обидва поля читають один байт → дані зсуваються | Перевірте: `FormatFlags` має `mp4:"1,size=8"`, `PCMSampleSize` має `mp4:"2,size=8"` |
| Неправильна інтерпретація `FormatFlags` | Integer читається як float → шум | Пам'ятайте: біт 0 (0x01) = Float, біт 2 (0x04) = Big-endian |
| Невідповідність `SampleSize` у `AudioSampleEntry` та `PCMSampleSize` | Зсув даних на 1-2 байти → десинхронізація | Завжди узгоджуйте: `audioEntry.SampleSize == pcmC.PCMSampleSize` |
| Ігнорування endianess | Big-endian дані читаються як little-endian → "кракозябри" звуку | Для вебу завжди використовуйте little-endian (`FormatFlags & 0x04 = 0`) |
| Float PCM без перевірки підтримки | Старі плеєри не відтворюють 32-біт float | Додайте валідацію: `if isFloat && !clientSupportsFloat { convertToInteger() }` |

## 📋 Чекліст для вашого проекту

```
[ ] При отриманні PCM-аудіо:
    • Шукайте `pcmC` бокс всередині `ipcm`/`fpcm` у `stsd`
    • Перевіряйте `FormatFlags & 0x08` для визначення float/integer
    • Логувайте `PCMSampleSize` для моніторингу якості

[ ] Для сумісності з вебом:
    • Integer PCM (`FormatFlags & 0x08 = 0`) підтримується всюди
    • 16-біт (`PCMSampleSize = 16`) — оптимальний баланс якості/бітрейту
    • Уникайте float PCM для широкого розповсюдження

[ ] При генерації нових сегментів:
    • Узгоджуйте `SampleSize` у `AudioSampleEntry` з `PCMSampleSize` у `pcmC`
    • Встановлюйте `FormatFlags` коректно: 0 для integer, 1 для float
    • Додавайте `pcmC` як вкладений бокс у `ipcm`/`fpcm`

[ ] Для дебагу:
    • Логуйте сирий вміст: log.Printf("🔊 pcmC: % x", pcmCBytes)
    • Перевіряйте відповідність: if audioEntry.SampleSize != pcmC.PCMSampleSize { ... }
    • Використовуйте `Stringify()` для людського виводу

[ ] Для тестування:
    • Напишіть round-trip тест: Marshal → Unmarshal → порівняння
    • Протестуйте на реальних плеєрах: VLC, hls.js, Safari, ExoPlayer
    • Перевірте відтворення з різними конфігураціями: 16-біт integer, 32-біт float
```
