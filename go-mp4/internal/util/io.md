# 🔤 `util`: Допоміжні функції для роботи з рядками у `go-mp4`

Це **мінімалістичний, але критичний модуль** бібліотеки `go-mp4`, який надає утиліти для роботи з **C-подібними рядками** (null-terminated strings) — фундамент для коректного парсингу та запису текстових полів у форматах MP4/ISOBMFF.

---

## 🎯 Коротка відповідь

> **Це "міст" між рядками Go та бінарним форматом**: він перетворює рядки у C-стилі (з нуль-термінатором `\0`) для запису у файл і навпаки — читає байти до першого `\0` для отримання рядка — що критично для полів типу `hdlr.Name`, `udta.metadata`, тощо.

---

## 🧱 Основні функції

### 🔹 `ReadString` — читання C-рядка з потоку

```go
func ReadString(r io.Reader) (string, error) {
    b := make([]byte, 1)           // 🔹 Буфер для одного байта
    buf := bytes.NewBuffer(nil)    // 🔹 Буфер для накопичення результату
    
    for {
        // 🔹 Крок 1: Читання одного байта
        if _, err := r.Read(b); err != nil {
            return "", err  // ❌ Помилка читання (напр. EOF)
        }
        
        // 🔹 Крок 2: Перевірка на нуль-термінатор
        if b[0] == 0 {
            return buf.String(), nil  // ✅ Кінець рядка
        }
        
        // 🔹 Крок 3: Додавання байта у результат
        buf.Write(b)
    }
}
```

**🎯 Призначення**: Читати байти з потоку **до першого нуль-байта (`\0`)** і повертати їх як рядок — стандартний формат для текстових полів у багатьох боксах (напр. `hdlr.Name`, `©nam`, `©ART`).

**🔄 Алгоритм:**
```
🔹 Вхід: потік байт [0x48, 0x65, 0x6c, 0x6c, 0x6f, 0x00, ...]
         ("Hello" + null-термінатор)
│
▼
🔹 Цикл:
   • Читання 0x48 ('H') → b[0]!=0 → додаємо у buf
   • Читання 0x65 ('e') → додаємо
   • Читання 0x6c ('l') → додаємо
   • Читання 0x6c ('l') → додаємо
   • Читання 0x6f ('o') → додаємо
   • Читання 0x00 ('\0') → b[0]==0 → повертаємо "Hello" ✅
│
▼
🔹 Вихід: рядок "Hello"
```

**🔢 Приклад використання:**
```
🔹 Бокс `hdlr` (Handler Reference):
   • Поле `Name`: C-string, наприклад "VideoHandle\0"
   • Виклик: name, _ := util.ReadString(r)
   • Результат: "VideoHandle" (без \0)

🔹 Бокс `©nam` (iTunes metadata):
   • Поле `Data`: UTF-8 рядок з нуль-термінатором
   • Виклик: title, _ := util.ReadString(r)
   • Результат: "My Video Title"
```

### 🔹 `WriteString` — запис C-рядка у потік

```go
func WriteString(w io.Writer, s string) error {
    // 🔹 Крок 1: Запис основної частини рядка
    if _, err := w.Write([]byte(s)); err != nil {
        return err  // ❌ Помилка запису
    }
    
    // 🔹 Крок 2: Запис нуль-термінатора
    if _, err := w.Write([]byte{0}); err != nil {
        return err
    }
    
    return nil  // ✅ Успіх
}
```

**🎯 Призначення**: Записати рядок у потік **з додаванням нуль-термінатора (`\0`)** на кінці — для сумісності з форматом, що очікує C-подібні рядки.

**🔄 Алгоритм:**
```
🔹 Вхід: рядок "Hello", io.Writer
│
▼
🔹 Крок 1: Перетворення рядка у байти
   • []byte("Hello") = [0x48, 0x65, 0x6c, 0x6c, 0x6f]
   • Запис у потік: 5 байт

🔹 Крок 2: Додавання нуль-термінатора
   • []byte{0} = [0x00]
   • Запис у потік: 1 байт

🔹 Вихід у потоці: [0x48, 0x65, 0x6c, 0x6c, 0x6f, 0x00] = "Hello\0" ✅
```

**🔢 Приклад використання:**
```
🔹 Створення боксу `hdlr`:
   • name := "VideoHandle"
   • util.WriteString(w, name)
   • Результат у файлі: "VideoHandle\0" (13 байт)

🔹 Створення iTunes metadata:
   • title := "My Video"
   • util.WriteString(w, title)
   • Результат: "My Video\0" (9 байт)
```

## 🔍 Чому це важливо для MP4?

### 🔹 Стандарт вимагає C-подібні рядки

Багато полів у специфікації ISO/IEC 14496-12 визначені як **null-terminated strings**:

| Бокс | Поле | Формат | Приклад |
|------|------|--------|---------|
| `hdlr` | `Name` | C-string | `"VideoHandle\0"` |
| `©nam` | `Data` | UTF-8 C-string | `"My Video\0"` |
| `©ART` | `Data` | UTF-8 C-string | `"Artist Name\0"` |
| `©cmt` | `Data` | UTF-8 C-string | `"Comment\0"` |

**🎯 Призначення**: Забезпечити **сумісність зі стандартом** — плеєри очікують нуль-термінатор для визначення кінця рядка.

---

### 🔹 Особливості кодування

```
🔹 ASCII vs UTF-8:
• Більшість старих боксів використовують ASCII
• iTunes metadata (`ilst`) використовують UTF-8 з префіксом типу

🔹 Нуль-термінатор:
• Завжди один байт: 0x00
• Не враховується у довжині рядка для користувача
• Обов'язковий для коректного парсингу

🔹 Порожній рядок:
• Вхід: "" → Вихід у файлі: [0x00] (тільки термінатор)
• Читання: [0x00] → Повертає "" (порожній рядок)
```

## 🔄 Порівняння з іншими підходами

| Підхід | Переваги | Недоліки | Коли використовувати |
|--------|----------|----------|---------------------|
| **`util.ReadString/WriteString`** | ✅ Простий, сумісний зі стандартом | ❌ Не підтримує довжину-префікс (Pascal) | Для стандартних боксів з C-string |
| **`string` тег у `mp4:"...,string"`** | ✅ Автоматичний, інтегрований у marshal/unmarshal | ❌ Менше контролю над кодуванням | Для більшості випадків у бібліотеці |
| **`string=c_p` тег (Pascal-style)** | ✅ Підтримує довжину-префікс | ❌ Не сумісний з усіма плеєрами | Для QuickTime-специфічних боксів |
| **Ручне читання/запис** | ✅ Повний контроль | ❌ Складніше, більше коду | Для екзотичних форматів |

## 🛠️ Практичне використання у вашому CCTV HLS Processor

### 🔹 Приклад 1: Читання метаданих з боксу `hdlr`

```go
// 🔹 Структура боксу hdlr (спрощено)
type Hdlr struct {
    mp4.FullBox
    PreDefined  uint32
    HandlerType uint32
    Reserved    [3]uint32
    Name        string  // 🔹 C-string
}

// 🔹 Ручний парсинг (якщо автоматичний не підходить)
func parseHdlrName(r io.Reader) (string, error) {
    // 🔹 Пропускаємо фіксовані поля (32 байти)
    _, err := io.CopyN(io.Discard, r, 32)
    if err != nil {
        return "", err
    }
    
    // 🔹 Читаємо Name як C-string
    name, err := util.ReadString(r)
    if err != nil {
        return "", err
    }
    
    return name, nil
}

// 🔹 Використання:
name, err := parseHdlrName(reader)
if err != nil {
    log.Printf("❌ Failed to read handler name: %v", err)
} else {
    log.Printf("✅ Handler: %s", name)  // "VideoHandle"
}
```

### 🔹 Приклад 3: Безпечне читання з обмеженням довжини

```go
// 🔹 Функція для читання рядка з максимальним розміром
func ReadStringWithLimit(r io.Reader, maxLen int) (string, error) {
    b := make([]byte, 1)
    buf := bytes.NewBuffer(nil)
    
    for buf.Len() < maxLen {
        if _, err := r.Read(b); err != nil {
            return "", err
        }
        if b[0] == 0 {
            return buf.String(), nil
        }
        buf.Write(b)
    }
    
    // 🔹 Якщо досягнуто ліміт без знаходження \0 → помилка
    return "", fmt.Errorf("string exceeds max length %d", maxLen)
}

// 🔹 Використання для валідації вхідних даних:
name, err := ReadStringWithLimit(reader, 256)
if err != nil {
    log.Printf("⚠️  Invalid name field: %v", err)
    // 🔹 Пропустити до кінця поля або відхилити файл
}
```

## ⚠️ Поширені помилки та як їх уникнути

| Помилка | Симптоми | Рішення |
|---------|----------|---------|
| Відсутність нуль-термінатора у файлі | `ReadString` читає до кінця файлу → помилка EOF | Перевіряйте вхідні файли: чи всі рядки закінчуються `\0` |
| Запис рядка без термінатора | Інші інструменти не можуть розпарсити поле | Завжди використовуйте `WriteString`, не `w.Write([]byte(s))` |
| Читання після кінця рядка | Зсув позиції, пошкодження наступних полів | `ReadString` зупиняється на `\0` — не читайте зайвого |
| Неправильне кодування (ASCII vs UTF-8) | "Кракозябри" у назвах з не-ASCII символами | Використовуйте UTF-8 для нових боксів, перевіряйте специфікацію |
| Порожній рядок обробляється некоректно | Очікується "" але отримується помилка | `ReadString` коректно повертає "" для [0x00] — перевіряйте логіку |

## 📋 Чекліст для вашого проекту

```
[ ] При читанні рядків з файлу:
    • Використовуйте ReadString для полів, визначених як C-string
    • Перевіряйте помилки: EOF може означати пошкоджений файл
    • Для безпеки: додавайте ліміт довжини через ReadStringWithLimit

[ ] При записі рядків у файл:
    • Завжди використовуйте WriteString, не забувайте про \0
    • Перевіряйте помилки запису: io.Writer може повернути error
    • Для сумісності: використовуйте UTF-8 для нових метаданих

[ ] Для валідації вхідних даних:
    • Перевіряйте довжину рядків перед записом (напр. макс. 255 символів)
    • Відхиляйте рядки з контрольними символами (окрім \0)
    • Логувайте попередження для не-ASCII символів у старих боксах

[ ] Для дебагу:
    • Логувайте прочитані рядки: log.Printf("📝 Read: %q", name)
    • Перевіряйте наявність \0 у вихідному файлі: hexdump -C file.mp4 | grep 00
    • Тестуйте з рядками різної довжини: "", "a", "Hello", дуже_довгий_рядок

[ ] Для тестування:
    • Створюйте тестові буфери з контрольованими C-рядками
    • Перевіряйте round-trip: WriteString → ReadString → порівняння
    • Тестуйте крайні випадки: порожній рядок, тільки \0, дуже довгі рядки
```
