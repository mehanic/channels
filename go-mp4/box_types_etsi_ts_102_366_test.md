## 🔍 Детальний розбір тест-кейсу: "dac3"

### ✅ Вхідні дані (`src`) — параметри аудіо

```go
src: &Dac3{
    Fscod:       0,           // 48 kHz частота дискретизації
    Bsid:        8,           // AC-3 стандартна версія бітстріму
    Acmod:       7,           // 3/2 = 5.1 каналів (L,C,R,Ls,Rs)
    LfeOn:       1,           // ✅ Є сабвуфер (LFE)
    BitRateCode: 0x7,         // 7 = ~192 kbps для 5.1
    // Bsmod=0 (за замовчуванням), Reserved=0 (const)
},
```

**📊 Що це означає для вашого стріму:**

| Параметр | Значення | Практичне значення |
|----------|----------|-------------------|
| `Fscod=0` | 48000 Hz | ✅ Стандарт для ТВ/стрімінгу |
| `Bsid=8` | AC-3 standard | Сумісність з усіма плеєрами |
| `Acmod=7` | 3/2 (5.1) | ✅ Об'ємний звук: L, C, R, Ls, Rs |
| `LfeOn=1` | LFE enabled | ✅ Сабвуфер для низьких частот |
| `BitRateCode=7` | ~192 kbps | Баланс якості та економії бітрейту |

> 🎯 **Висновок**: Це конфігурація для **5.1 аудіо з сабвуфером**, оптимізована для трансляції новин/фільмів.

### ✅ Очікувані байти (`bin`) — бітова упаковка

```go
bin: []byte{0x10, 0x3c, 0xe0},  // Рівно 3 байти!
```

**🔢 Детальна розбивка бітової упаковки:**

```
📦 Структура полів (всього 24 біти = 3 байти):
┌─────────────────────────────────┐
│ Поле          │ Біти │ Значення │
├─────────────────────────────────┤
│ Fscod         │ 2    │ 0 = 00   │
│ Bsid          │ 5    │ 8 = 01000│
│ Bsmod         │ 3    │ 0 = 000  │
│ Acmod         │ 3    │ 7 = 111  │
│ LfeOn         │ 1    │ 1 = 1    │
│ BitRateCode   │ 5    │ 7 = 00111│
│ Reserved      │ 5    │ 0 = 00000│
└─────────────────────────────────┘
```

**🧮 Покрокова упаковка у байти:**

```
🔹 БАЙТ 0: [Fscod:2][Bsid:5][Bsmod:1] = 8 біт
   00 + 01000 + 0 = 0001 0000 = 0x10 ✅

🔹 БАЙТ 1: [Bsmod:2][Acmod:3][LfeOn:1][BitRate:2] = 8 біт  
   00 + 111 + 1 + 00 = 0011 1100 = 0x3C ✅

🔹 БАЙТ 2: [BitRate:3][Reserved:5] = 8 біт
   111 + 00000 = 1110 0000 = 0xE0 ✅

🎯 Результат: [0x10, 0x3C, 0xE0] — саме те, що в тесті!
```

> 🎯 **Магія бібліотеки**: Ви працюєте з окремими полями (`Fscod`, `Acmod`), а бібліотека сама пакує їх у біти за тегами `mp4:"...,size=N"`.

## 🔄 Чотири операції, що тестуються

### 🔹 1. `Marshal` — серіалізація (структура → 3 байти)

```go
buf := bytes.NewBuffer(nil)
n, err := Marshal(buf, tc.src, tc.ctx)
require.NoError(t, err)
assert.Equal(t, uint64(len(tc.bin)), n)  // має бути рівно 3 байти!
assert.Equal(t, tc.bin, buf.Bytes())      // байт в байт: [0x10, 0x3c, 0xe0]
```

**Що перевіряємо:**
```
📦 Вхід: Dac3{Fscod:0, Bsid:8, Acmod:7, LfeOn:1, BitRateCode:7}
              │
              ▼
         Marshal() + бітова упаковка (2+5+3+3+1+5+5=24 біти)
              │
              ▼
📤 Вихід: []byte{0x10, 0x3c, 0xe0}  ← рівно 3 байти!
              │
              ▼
✅ Порівнюємо з очікуваним — кожен біт на своєму місці
```

---

### 🔹 2. `Unmarshal` — десеріалізація (3 байти → структура)

```go
r := bytes.NewReader(tc.bin)
n, err = Unmarshal(r, uint64(len(tc.bin)), tc.dst, tc.ctx)
require.NoError(t, err)
assert.Equal(t, uint64(buf.Len()), n)  // прочитано рівно 3 байти
assert.Equal(t, tc.src, tc.dst)         // структура відновлена точно!
```

**🔁 Round-trip тест — найважливіша перевірка:**
```
Структура → 3 байти → Структура'

Якщо Структура == Структура' → ✅ бітова упаковка ідемпотентна
Якщо ні → ❌ втрачено біт → декодер отримає хибні параметри → звук "зламається"
```

---

### 🔹 3. `UnmarshalAny` — динамічний парсинг за типом

```go
dst, n, err := UnmarshalAny(
    bytes.NewReader(tc.bin), 
    tc.src.GetType(),        // BoxTypeDAC3() = "dac3"
    uint64(len(tc.bin)), 
    tc.ctx,
)
require.NoError(t, err)
assert.Equal(t, tc.src, dst)
```

**Навіщо `UnmarshalAny`?**
```
🔍 Сценарій: Ви читаєте файл і не знаєте наперед, який бокс зустріне наступним

1. ReadBoxInfo() → тип="dac3", size=3
2. UnmarshalAny(r, "dac3", 3, ctx) 
3. Бібліотека сама:
   • Знаходить зареєстровану структуру для "dac3" (Dac3{})
   • Створює її екземпляр
   • Розпаковує 24 біти у 7 полів
   • Повертає готовий об'єкт

✅ Ви не пишете 50+ `case "dac3":` у своєму коді!
```

## 🛠️ Додаткові перевірки: позиція читача

```go
// Після Unmarshal: перевіряємо, що курсор в кінці прочитаних даних
s, err := r.Seek(0, io.SeekCurrent)
require.NoError(t, err)
assert.Equal(t, int64(buf.Len()), s)  // прочитано рівно 3 байти
```

**Чому це критично?**
```
📦 Файл: [moof][traf][dac3:3B][trun][mdat]
                    │
                    ▼
              Парсимо "dac3" (рівно 3 байти)
                    │
                    ▼
❌ Якщо Unmarshal прочитає 4 байти → "trun" зсунутий → таймстемпи зламались
✅ Якщо курсор точно в кінці "dac3" → наступний бокс читається коректно
```

## 🎯 Чому цей тест критичний для вашого HLS-процесора?

### 🔹 Сценарій 1: Коректне відтворення 5.1 аудіо

```
📡 Камера надсилає fMP4 з AC-3 5.1:
1. Unmarshal читає dac3: Acmod=7, LfeOn=1
2. Ви передаєте ці параметри у HLS-плейлист:
   #EXT-X-MEDIA:CHANNELS="6"  ← 5 каналів + LFE = 6
3. Плеєр (Safari, VLC) ініціалізує декодер з правильними налаштуваннями

❌ Без тесту: помилка в бітовій упаковці → Acmod=7 читається як 3
   → плеєр думає, що це 3.0 звук → сабвуфер мовчить, канали переплутані
```

### 🔹 Сценарій 2: Економія бітрейту без втрати якості

```
📝 Ви хочете оптимізувати стрімінг для мобільних:
1. Змінюєте BitRateCode з 7 (192 kbps) → 4 (96 kbps) у dac3
2. Записуєте оновлений бокс у новий сегмент
3. Клієнт отримує легший потік, але звук залишається чітким

✅ Тест гарантує, що 5 біт `BitRateCode` пакуються/розпаковуються коректно
❌ Без тесту: зсув на 1 біт → бітрейт 96 kbps читається як 256 kbps
   → клієнт не може декодувати → звук обривається
```

### 🔹 Сценарій 3: Підтримка спеціальних аудіо-доріжок

```
♿ Ви додаєте аудіо-опис для слабозорих:
1. Встановлюєте Bsmod=2 (visually impaired) у dac3
2. Додаєте мітку у HLS: CHARACTERISTICS="public.accessibility.describes-video"
3. Плеєр показує іконку 🎧 і дозволяє вибрати цю доріжку

✅ Тест гарантує, що 3 біти `Bsmod` не "з'їжджають" при серіалізації
❌ Без тесту: Bsmod=2 читається як 0 → спеціальна доріжка не відображається
```

## ⚠️ Поширені помилки та як їх уникнути

| Помилка | Симптоми | Рішення |
|---------|----------|---------|
| Неправильний порядок полів у структурі | Біти "з'їжджають", парсинг ламається | Дотримуйтесь порядку `mp4:"0", mp4:"1", ...` |
| `Reserved != 0` | Декодер відмовляє читати бокс | Завжди встановлюйте `Reserved=0` (тег `const=0`) |
| `LfeOn=1` без `Acmod>=5` | Сабвуфер не працює або артефакти | Вмикайте LFE тільки для 3.1/5.1/7.1 |
| Ігнорування бітового вирівнювання | Поля читаються зі зсувом на 1 біт | Пам'ятайте: 2+5+3+3+1+5+5 = 24 біти = 3 байти |
| Неправильне кодування `BitRateCode` | Звук тихий або "захлинається" | Використовуйте таблицю бітрейтів з стандарту ETSI |
