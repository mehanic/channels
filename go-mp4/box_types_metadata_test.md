# 🧪 Тест `TestBoxTypesMetadata`: iTunes/QuickTime метадані у MP4

Це **інтеграційний тест** для бібліотеки `go-mp4`, який перевіряє коректну роботу **серіалізації/десеріалізації** боксів метаданих у стилі **iTunes/QuickTime** (`ilst`, `data`, `keys` тощо).

---

## 🎯 Коротка відповідь

> **Цей тест гарантує, що метадані (назва, виконавець, обкладинка, користувацькі поля) коректно вбудовуються у MP4-файли та читаються плеєрами** — критично для підтримки медіа-бібліотек (iTunes, Plex, VLC) та відображення інформації про контент.

```go
func TestBoxTypesMetadata(t *testing.T) {
    // 1. Масив тест-кейсів (14 кейсів для різних типів метаданих)
    testCases := []struct {

    // 2. Запуск кожного кейсу
    for _, tc := range testCases {
        t.Run(tc.name, func(t *testing.T) {
            // Тестуємо 4 операції: Marshal → Unmarshal → UnmarshalAny → Stringify
        })
    }
}
```

### 🔹 Кейси 3-11: `data` бокс з різними `DataType`

| Кейс | DataType | Значення | Очікуваний `str` |
|------|----------|----------|-----------------|
| `ilst data (binary)` | `0` | 🔹 Сирі байти | `Data=[0x66, 0x6f, 0x6f]` |
| `ilst data (utf8)` | `1` | 🔹 UTF-8 текст | `Data="foo"` |
| `ilst data (utf8 escaped)` | `1` | Текст з недрюкованими символами | `Data=".foo"` (0x00 → `.`) |
| `ilst data (utf16)` | `2` | UTF-16 текст | `Data=[0x66, 0x6f, 0x6f]` |
| `ilst data (mac string)` | `3` | MacRoman текст | `Data=[0x66, 0x6f, 0x6f]` |
| `ilst data (jpsg)` | `14` | 🔹 JPEG зображення | `Data=[0x66, 0x6f, 0x6f]` |
| `ilst data (int)` | `21` | 🔹 Ціле число | `Data=[0x66, 0x6f, 0x6f]` |
| `ilst data (float32)` | `22` | Float 32-bit | `Data=[0x66, 0x6f, 0x6f]` |
| `ilst data (float64)` | `23` | Float 64-bit | `Data=[0x66, 0x6f, 0x6f]` |

**🔢 Структура `Data` боксу (12 байт заголовок + дані):**
```
📦 Data бокс:
┌─────────────────────────────────┐
│ Байти 0-3: DataType (uint32)    │ ← 0=BINARY, 1=UTF8, 21=INT...
├─────────────────────────────────┤
│ Байти 4-7: DataLang (uint32)    │ ← мова (рідко використовується)
├─────────────────────────────────┤
│ Байти 8+:   Data ([]byte)       │ ← сирі байти значення
└─────────────────────────────────┘
```

**🎯 Ключова особливість `Stringify`**:
```go
// Для DataType=1 (UTF8):
case DataTypeStringUTF8:
    return fmt.Sprintf("\"%s\"", util.EscapeUnprintables(string(data.Data))), true
// 🔹 Результат: "foo" замість [0x66, 0x6f, 0x6f]

// Для недрюкованих символів (0x00):
util.EscapeUnprintables("\x00foo") → ".foo"  // 0x00 замінюється на '.'
```

```
📦 Key запис (змінна довжина):
┌─────────────────────────────────┐

├─────────────────────────────────┤
│ Байти 4-7:   KeyNamespace       │ ← завжди 4 байти ("mdta", "com."...)
├─────────────────────────────────┤
│ Байти 8+:    KeyValue           │ ← решта назви ключа (KeySize - 8 байт)
└─────────────────────────────────┘
```

**🎯 `GetFieldLength` для динамічного поля:**
```go
func (k *Key) GetFieldLength(name string, ctx Context) uint {
    switch name {
    case "KeyValue":
        // 🔹 Довжина KeyValue = загальний розмір - 8 байт заголовка
        return uint(k.KeySize) - 8
    }
    panic(fmt.Errorf("invalid field: %s", name))
}
```

## 🔄 Чотири операції, що тестуються

### 🔹 1. `Marshal` — серіалізація (структура → байти)
```go
buf := bytes.NewBuffer(nil)
n, err := Marshal(buf, tc.src, tc.ctx)
assert.Equal(t, tc.bin, buf.Bytes())  // байт в байт!
```
**Перевіряє**: Чи коректно формуються байти з урахуванням `DataType`, контексту, динамічних полів.

---

### 🔹 2. `Unmarshal` — десеріалізація (байти → структура)
```go
r := bytes.NewReader(tc.bin)
n, err = Unmarshal(r, uint64(len(tc.bin)), tc.dst, tc.ctx)
assert.Equal(t, tc.src, tc.dst)  // 🔁 round-trip: ідемпотентність!
```
**Перевіряє**: Чи відновлюється структура точно такою ж після читання, з урахуванням контексту та динамічних полів.

---

### 🔹 3. `UnmarshalAny` — динамічний парсинг за типом
```go
dst, n, err := UnmarshalAny(bytes.NewReader(tc.bin), tc.src.GetType(), ...)
assert.Equal(t, tc.src, dst)
```
**Перевіряє**: Чи працює універсальний парсер для невідомих наперед типів боксів (`©nam`, `covr`, `----` тощо).

---

### 🔹 4. `Stringify` — людино-читабельний вивід
```go
str, err := Stringify(tc.src, tc.ctx)
assert.Equal(t, tc.str, str)
```
**Перевіряє**: Чи коректно формуються рядки для дебагу, зокрема:
- ✅ Перетворення `DataType` на читабельні назви: `1` → `"UTF8"`
- ✅ Екранування недрюкованих символів: `0x00` → `.`
- ✅ Форматування тексту в лапках: `"foo"` замість `[0x66, 0x6f, 0x6f]`

## ⚠️ Поширені помилки та як їх уникнути

| Помилка | Симптоми | Рішення |
|---------|----------|---------|
| Неправильний `DataType` | Текст відображається як "кракозябри" | Завжди використовуйте `DataTypeStringUTF8` (1) для тексту |
| Ігнорування контексту `UnderIlstMeta` | `data` бокс парситься неправильно | Завжди передавайте `ctx` з правильними прапорцями |
| Неправильне кодування тексту | Кирилиця/емодзі → "????????" | Використовуйте UTF-8 без BOM, перевіряйте `utf8.Valid()` |
| Переповнення `KeySize` у `keys` | Зсув даних → пошкодження файлу | Завжди `KeySize = 8 + len(KeyValue)` |
| Відсутній `data` бокс у метаданому | Метадане ігнорується плеєром | Завжди додавайте `Data` бокс всередині контейнера |
| Неправильне екранування у `Stringify` | Логи з недрюкованими символами важко читати | Використовуйте `util.EscapeUnprintables()` |
