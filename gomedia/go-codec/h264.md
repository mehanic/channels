# 🎬 Глибокий розбір `codec/h264.go` — парсинг H.264 бітстріму

Це **низькорівневий парсер стандарту ITU-T H.264/AVC**, який перетворює сирі біти відеопотоку на структуровані дані для вашого HLS-процесора. Розберемо архітектурно:

---

## 🧱 1. Архітектура: бітовий потік → структури → утиліти

```
[Raw H.264 NALU] 
       ↓
[BitStream] ← абстракція для читання біт-полів (не байтів!)
       ↓
[H264NaluHdr / SPS / PPS / SEI] ← Decode() парсить за специфікацією
       ↓
[Helper funcs] ← GetResolution, CreateAVCC, ConvertAnnexB↔AVCC
```

### 🔑 Ключова концепція: `BitStream`

H.264 використовує **експоненційні кодування Голомба** (UE/SE) та поля довільної довжини (1, 2, 5, 8 біт). Звичайне `binary.Read()` не працює.

```go
// Приклад: парсинг 5-бітового поля nal_unit_type
hdr.Nal_unit_type = bs.Uint8(5)  // читає рівно 5 біт, не цілий байт!

// Приклад: кодування Голомба (UE = unsigned exponential)
sh.Slice_type = bs.ReadUE()  // 0→1біт, 1→3біти, 2→3біти, 3→5біт...
```

> 💡 **Чому це важливо для HLS**: без коректного бітового парсингу ви не зможете витягти `log2_max_frame_num_minus4` з SPS → не зможете правильно інтерпретувати `frame_num` у слайсах → не зможете детектувати пропуски кадрів.

## 📦 2. Структури даних: відображення стандарту H.264

### 🎯 `H264NaluHdr` — заголовок NAL Unit (7.3.1 специфікації)

```go
type H264NaluHdr struct {
    Forbidden_zero_bit uint8  // 1 біт: має бути 0 (перевірка цілісності)
    Nal_ref_idc        uint8  // 2 біти: пріоритет для декодування (0-3)
    Nal_unit_type      uint8  // 5 біт: тип (1=P-slice, 5=I-slice, 7=SPS...)
}
```

| Поле | Практичне значення у вашому пайплайні |
|------|----------------------------------------|
| `Nal_ref_idc > 0` | Кадр містить референси → не можна відкидати при backpressure |
| `Nal_unit_type == 5` | I-frame → точка старту нового сегменту в `segmentAssembler` |
| `Nal_unit_type == 7/8` | SPS/PPS → мають бути в кожному HLS-сегменті для FFmpeg |

---

### 🗝️ `SPS` (Sequence Parameter Set) — "паспорт" відеопотоку

Це **найкритичніша структура** для вашого HLS Processor:

```go
type SPS struct {
    Profile_idc             uint8   // 66=Baseline, 77=Main, 100=High
    Level_idc               uint8   // 30=720p30, 41=1080p30, 51=4K
    Pic_width_in_mbs_minus1 uint64  // ширина в макроблоках (16px) - 1
    Pic_height_in_map_units_minus1 uint64 // висота
    Frame_mbs_only_flag     uint8   // 1=progressive, 0=interlaced
    Frame_cropping_flag     uint8   // чи є crop offsets
    // ... + VUI parameters (таймінг, колір, HRD)
}
```

#### 🔧 Функція `GetH264Resolution` — як це працює:

```go
// Формула з коментаря у коді (ITU-T H.264 Annex E):
width = ((pic_width_in_mbs_minus1 + 1) * 16) - 2*(crop_left + crop_right)
height = ((2 - frame_mbs_only_flag) * (pic_height_in_map_units_minus1 + 1) * 16) - 2*(crop_top + crop_bottom)
```

**Приклад розрахунку** для 1920×1080:
```
pic_width_in_mbs_minus1 = 119  → (119+1)*16 = 1920 ✓
pic_height_in_map_units_minus1 = 67 → (67+1)*16 = 1088
frame_mbs_only_flag = 1 → 1*1088 = 1088
crop_top+bottom = 4 → 1088 - 2*4 = 1080 ✓
```

> ⚠️ **Пастка**: у вашому коді є баг у розрахунку висоти:
> ```go
> heightCrop := uint32(s.Frame_crop_bottom_offset)*2 - uint32(s.Frame_crop_top_offset)*2
> // Має бути +, а не -:
> heightCrop := uint32(s.Frame_crop_top_offset + s.Frame_crop_bottom_offset) * 2
> ```

### 🧩 `PPS` (Picture Parameter Set) — налаштування кадру

```go
type PPS struct {
    Pic_parameter_set_id     uint64  // ID цього PPS (посилання з слайсів)
    Seq_parameter_set_id     uint64  // Посилання на SPS (критично для валідації!)
    Entropy_coding_mode_flag uint8   // 0=CAVLC, 1=CABAC (впливає на продуктивність)
}
```

**Чому це важливо**: якщо `PPS.Seq_parameter_set_id` не співпадає з `SPS.Seq_parameter_set_id` → відео не декодується. Ваш `segmentFinalizer` має це перевіряти.

---

### ℹ️ `SEI` (Supplemental Enhancement Information) — метадані

```go
type SEI struct {
    PayloadType uint16  // 5=unregistered user data, 1=timecode, 137=HDR...
    PayloadSize uint16
    Sei_payload SEIReaderWriter
}
```

#### 🎯 Практичне застосування: `UserDataUnregistered` (payload type 5)

Це **ключ до синхронізації субтитрів з відео**! Камери/енкодери можуть вставляти:
```
SEI payload type=5:
├─ UUID: 16 байт (наприклад, "com.alarabiya.cctv.sync")
└─ UserData: JSON з {seq: 12345, timestamp: 1712345678}
```

Ваш `segmentAssembler` може читати це для точної прив'язки субтитрів до відео-сегментів.

## 🔄 3. Конвертація форматів: Annex-B ↔ AVCC

### 📦 Два формати подання H.264:

| Формат | Де використовується | Структура |
|--------|---------------------|-----------|
| **Annex-B** | RTP, MPEG-TS, raw stream | `[0x00000001][NALU]` або `[0x000001][NALU]` |
| **AVCC** (ISO/IEC 14496-15) | MP4, fMP4, HLS segments | `[4-byte length][NALU]` без start codes |

### 🔧 Функції конвертації:

```go
// Annex-B → AVCC (для запису у fMP4 сегмент)
func ConvertAnnexBToAVCC(annexb []byte) []byte {
    // Замінює 0x00000001 на 4-byte big-endian length
    binary.BigEndian.PutUint32(avcc, uint32(len(annexb)-4))
    return avcc
}

// AVCC → Annex-B (для відправки у WebSocket або FFmpeg pipe)
func CovertAVCCToAnnexB(avcc []byte) {
    avcc[0]=0x00; avcc[1]=0x00; avcc[2]=0x00; avcc[3]=0x01
}
```

> 💡 **У вашому пайплайні**: `segmentFinalizer` отримує fMP4 з AVCC-форматом, але `createTSSegment` через FFmpeg може вимагати Annex-B. Ці функції — міст між ними.

## 🧪 4. `CreateH264AVCCExtradata` — генерація `avcC` для fMP4

Це **найскладніша функція**, яка створює `AVCDecoderConfigurationRecord` (специфікація ISO/IEC 14496-15, розділ 5.2.4.1.1):

```go
func CreateH264AVCCExtradata(spss [][]byte, ppss [][]byte) ([]byte, error)
```

### 📐 Структура `avcC` (extradata):

```
Offset  Size  Field
0       1     configurationVersion (always 1)
1       1     AVCProfileIndication (з SPS[1])
2       1     profile_compatibility (з SPS[2])
3       1     AVCLevelIndication (з SPS[3])
4       1     reserved (0xFF) + lengthSizeMinusOne (0x03 = 4-byte length)
5       1     numOfSequenceParameterSets (зазвичай 1)
6-7     2     sequenceParameterSetLength (big-endian)
8-N     N     SPS NALU (без start code!)
...     ...   PPS аналогічно
```

### 🎯 Навіщо це у HLS?

Коли ви створюєте fMP4-ініціалізаційний сегмент (`init.mp4`), поле `avcC` у треку `hvc1/avc1` **має містити SPS/PPS**. Без цього:
- FFmpeg видасть `missing parameter sets`
- Браузери не зможуть ініціалізувати MSE SourceBuffer
- VLC відмовиться відтворювати

## ⚙️ 5. `H264VuiParameters` — Video Usability Information

Це **додаткові метадані**, які впливають на відтворення:

```go
type H264VuiParameters struct {
    // Aspect Ratio
    AspectRatioInfoPresentFlag uint8
    AspectRatioIdc             uint8  // 1=1:1, 2=4:3, 3=16:9, 255=extended
    
    // Timing (критично для PTS/DTS!)
    TimingInfoPresentFlag      uint8
    NumUnitsInTick             uint32 // наприклад, 1 для 90kHz clock
    TimeScale                  uint32 // наприклад, 90000 для 30fps: 90000/30=3000 units/frame
    FixedFrameRateFlag         uint8  // 1=constant framerate
    
    // Colour (для HDR/WCG)
    ColourDescriptionPresentFlag uint8
    ColourPrimaries              uint8  // 1=BT.709, 9=BT.2020
    TransferCharacteristics      uint8  // 1=SDR, 16=PQ, 18=HLG
    MatrixCoefficients           uint8  // 1=BT.709, 9=BT.2020
}
```

### 🎯 Практичне застосування:

1. **PTS розрахунок**: `TimeScale / NumUnitsInTick = clock frequency` → конвертація у 90kHz для HLS.
2. **Display aspect ratio**: якщо `AspectRatioIdc=3` (16:9), а пікселі квадратні → не потрібно stretch.
3. **HDR signaling**: `TransferCharacteristics=16` (PQ) → клієнт має увімкнути HDR mode.

> ⚠️ **У вашому коді**: є TODO-коментар про проблеми з парсингом `PicStructPresentFlag`. Це тому, що ці поля **умовно обов'язкові** — залежать від `profile_idc` та `level_idc`. Безпечніше парсити їх у окремому `if BitstreamRestrictionFlag == 1`.

## 🐞 6. Потенційні баги та покращення

### ❗ Критичні:

1. **Баг у `GetH264Resolution`** (вже згадувався):
   ```go
   // НЕПРАВИЛЬНО:
   heightCrop := uint32(s.Frame_crop_bottom_offset)*2 - uint32(s.Frame_crop_top_offset)*2
   // ПРАВИЛЬНО:
   heightCrop := (uint32(s.Frame_crop_top_offset) + uint32(s.Frame_crop_bottom_offset)) * 2
   ```

2. **Перезапис `SarWidth` у `VuiParameters.Decode`**:
   ```go
   h264Vui.SarWidth = bs.Uint16(16)  // рядок 1
   h264Vui.SarWidth = bs.Uint16(16)  // рядок 2 ← має бути SarHeight!
   ```

3. **Неініціалізований слайс у `SPS`**:
   ```go
   sps.Offset_for_ref_frame[i] = bs.ReadSE()  // panic якщо слайс не ініціалізовано!
   // Треба: sps.Offset_for_ref_frame = make([]int64, num_ref_frames...)
   ```

### 💡 Покращення для вашого пайплайну:

```go
// 1. Додати валідацію після Decode()
func (sps *SPS) Validate() error {
    if sps.Profile_idc == 0 || sps.Level_idc == 0 {
        return errors.New("invalid SPS: zero profile/level")
    }
    if sps.Pic_width_in_mbs_minus1 > 383 { // Max 4K: (384*16)-crop
        return errors.New("resolution too high")
    }
    return nil
}

// 2. Кешування розпарсених SPS/PPS за ID
type ParamSetCache struct {
    spsMap map[uint64]*SPS  // seq_parameter_set_id → SPS
    ppsMap map[uint64]*PPS  // pic_parameter_set_id → PPS
    mu     sync.RWMutex
}

// 3. Helper для детекції ключових кадрів
func (hdr *H264NaluHdr) IsKeyFrame() bool {
    return hdr.Nal_unit_type == 5 || // H.264 IDR
           hdr.Nal_unit_type == 19 || hdr.Nal_unit_type == 20 // H.265 IDR
}
```

## 🎯 7. Інтеграція з вашим CCTV HLS Processor

### 📍 У `segmentAssembler`:

```go
func (sa *SegmentAssembler) processH264NALU(nalu []byte) {
    bs := codec.NewBitStream(nalu[1:]) // пропустити перший байт заголовка
    var hdr codec.H264NaluHdr
    hdr.Decode(bs)
    
    switch hdr.Nal_unit_type {
    case codec.H264_NAL_SPS:
        var sps codec.SPS
        sps.Decode(codec.NewBitStream(codec.CovertRbspToSodb(nalu[1:])))
        sa.currentResolution = sps.GetResolution() // зберегти для метрик
        sa.spsCache[sps.Seq_parameter_set_id] = &sps
        
    case codec.H264_NAL_I_SLICE:
        if sa.shouldStartNewSegment() {
            sa.finalizeCurrentSegment() // закрити попередній .ts
            sa.startNewSegment(hdr.Frame_num) // новий сегмент з цього I-frame
        }
        
    case codec.H264_NAL_SEI:
        var sei codec.SEI
        sei.Decode(bs)
        if sei.PayloadType == 5 { // user data
            if syncData, ok := sei.Sei_payload.(*codec.UserDataUnregistered); ok {
                sa.syncSubtitleWithVideo(syncData.UserData) // прив'язка субтитрів!
            }
        }
    }
}
```

### 📍 У `createTSSegment`:

```go
// Перед запуском ffprobe - перевірка валідності сегменту
func validateH264Segment(nalus [][]byte) error {
    hasSPS, hasPPS, hasIDR := false, false, false
    for _, nalu := range nalus {
        bs := codec.NewBitStream(nalu)
        var hdr codec.H264NaluHdr
        hdr.Decode(bs)
        
        switch hdr.Nal_unit_type {
        case codec.H264_NAL_SPS: hasSPS = true
        case codec.H264_NAL_PPS: hasPPS = true
        case codec.H264_NAL_I_SLICE: hasIDR = true
        }
    }
    
    if !hasSPS || !hasPPS {
        return errors.New("missing parameter sets - segment not decodable")
    }
    if !hasIDR {
        logger.Warn("segment without IDR - may cause decoding issues")
    }
    return nil
}
```

## 🧭 Висновок: навіщо ця складність?

| Рівень | Що дає цей код |
|--------|----------------|
| **Бітовий** | Коректний парсинг експоненційних кодів Голомба → точне читання `frame_num`, `poc` |
| **Структурний** | Типізовані SPS/PPS → валідація сумісності параметрів між сегментами |
| **Контейнерний** | Annex-B ↔ AVCC конвертація → сумісність fMP4 ↔ HLS ↔ FFmpeg |
| **Метадані** | VUI/SEI парсинг → синхронізація субтитрів, HDR signaling, timing correction |
| **Продуктивність** | Бітовий парсинг у пам'яті (без виклику зовнішніх процесів) → low-latency обробка |

> 🔑 **Головна ідея**: цей код — **декодер протоколу**, а не декодер відео. Він не відтворює картинку, але дає вашому пайплайну **семантичне розуміння** бітстріму: де ключові кадри, які параметри декодування, як синхронізувати з аудіо/субтитрами.

Без нього `segmentAssembler` працював би "наосліп", розрізаючи потік довільно → сегменти без SPS/PPS → FFmpeg помиляється → HLS-плейлист пошкоджений → клієнти не можуть відтворювати.

💡 **Порада**: додайте юніт-тести на `GetH264Resolution` з реальними SPS з ваших камер — це вбереже від багів з кропом та interlaced-відео.
