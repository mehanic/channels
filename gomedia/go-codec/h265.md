# 🎬 Глибокий розбір `codec/h265.go` — парсинг H.265/HEVC бітстріму

Це **повноцінна реалізація парсера стандарту ITU-T H.265/HEVC**, який розширює ваш CCTV HLS Processor підтримкою сучасного кодека. Розберемо архітектурно, шар за шаром:

---

## 🧱 1. Архітектура: чим H.265 відрізняється від H.264

```
[H.264]                          [H.265/HEVC]
─────────────────────────────────────────────────
NAL header: 1 байт (8 біт)   →   NAL header: 2 байти (16 біт)
├─ forbidden_zero_bit: 1          ├─ forbidden_zero_bit: 1
├─ nal_ref_idc: 2                 ├─ nal_unit_type: 6 ← розширено!
├─ nal_unit_type: 5               ├─ nuh_layer_id: 6 ← нове!
                                  └─ nuh_temporal_id_plus1: 3 ← нове!

Parameter Sets:
├─ SPS + PPS                   →   VPS + SPS + PPS ← додано VPS!
├─ Макроблоки 16×16            →   CTU (Coding Tree Units) 8×8–64×64
├─ 9 типів слайсів             →   20+ типів слайсів з кращою компресією
```

### 🔑 Ключова структура: `H265NaluHdr`

```go
type H265NaluHdr struct {
    Forbidden_zero_bit    uint8  // 1 біт: має бути 0
    Nal_unit_type         uint8  // 6 біт: 0-63 (проти 5 біт у H.264)
    Nuh_layer_id          uint8  // 6 біт: для scalable/multi-layer coding
    Nuh_temporal_id_plus1 uint8  // 3 біти: temporal scalability (0-7)
}

func (hdr *H265NaluHdr) Decode(bs *BitStream) {
    hdr.Forbidden_zero_bit = bs.GetBit()           // 1 біт
    hdr.Nal_unit_type = bs.Uint8(6)                // 6 біт
    hdr.Nuh_layer_id = bs.Uint8(6)                 // 6 біт
    hdr.Nuh_temporal_id_plus1 = bs.Uint8(3)        // 3 біти
    // Разом: 1+6+6+3 = 16 біт = 2 байти ✓
}
```

> 💡 **Практичне значення**: `nuh_temporal_id_plus1` дозволяє вашому `segmentAssembler` фільтрувати кадри за часовим шаром — корисно для адаптивного стрімінгу (наприклад, відкидати високочастотні шари при низькій пропускній здатності).

## 🗝️ 2. VPS (Video Parameter Set) — новий рівень абстракції

### 🎯 Навіщо потрібен VPS?

У H.264 всі параметри були в SPS. У H.265 додано **ієрархію параметрів**:

```
VPS (Video Parameter Set)
├─ Загальні налаштування для всього потоку
├─ Підтримка multi-layer coding (SVC/MVC)
├─ Timing information для всіх шарів
└─ Профіль/рівень (ProfileTierLevel)

SPS (Sequence Parameter Set)
├─ Посилання на VPS: sps_video_parameter_set_id
├─ Роздільна здатність, колір, бітова глибина
└─ Параметри кодування послідовності

PPS (Picture Parameter Set)
├─ Посилання на SPS: pps_seq_parameter_set_id
└─ Налаштування окремого кадру/слайсу
```

### 🔧 Структура `VPS` — критичні поля:

```go
type VPS struct {
    Vps_video_parameter_set_id    uint8  // ID цього VPS (0-15)
    Vps_max_layers_minus1         uint8  // Макс. кількість шарів - 1
    Vps_max_sub_layers_minus1     uint8  // Макс. temporal layers - 1
    
    Ptl ProfileTierLevel  // Профіль/рівень (як у H.264, але розширений)
    
    // Timing info для синхронізації
    Vps_timing_info_present_flag uint8
    TimeInfo VPSTimeInfo       // num_units_in_tick, time_scale → fps розрахунок
    
    // HRD parameters для buffer management
    Vps_num_hrd_parameters uint64
}
```

### 🎯 Практичне застосування у вашому пайплайні:

```go
// У segmentAssembler: перевірка сумісності параметрів
func (sa *SegmentAssembler) validateH265ParamSets(vps, sps, pps []byte) error {
    var rawVPS codec.VPS
    rawVPS.Decode(vps)
    
    var rawSPS codec.H265RawSPS
    rawSPS.Decode(sps)
    
    // Критична перевірка: SPS має посилатися на валідний VPS
    if rawSPS.Sps_video_parameter_set_id != rawVPS.Vps_video_parameter_set_id {
        return fmt.Errorf("SPS references VPS ID %d, but got %d", 
            rawSPS.Sps_video_parameter_set_id, 
            rawVPS.Vps_video_parameter_set_id)
    }
    
    // Розрахунок FPS для PTS нормалізації
    if rawVPS.TimeInfo.Vps_time_scale > 0 {
        fps := float64(rawVPS.TimeInfo.Vps_time_scale) / 
               float64(rawVPS.TimeInfo.Vps_num_units_in_tick)
        sa.currentFPS = fps
    }
    
    return nil
}
```

## 📐 3. H.265 SPS — роздільна здатність та параметри кодування

### 🔍 Ключові відмінності від H.264 SPS:

```go
type H265RawSPS struct {
    // Посилання на VPS
    Sps_video_parameter_set_id uint8
    
    // Роздільна здатність у пікселях (не макроблоках!)
    Pic_width_in_luma_samples  uint64  // ← прямо ширина у пікселях
    Pic_height_in_luma_samples uint64  // ← прямо висота у пікселях
    
    // Conformance window (аналог crop у H.264)
    Conformance_window_flag    uint8
    Conf_win_left_offset       uint64
    Conf_win_right_offset      uint64
    Conf_win_top_offset        uint64
    Conf_win_bottom_offset     uint64
    
    // Бітова глибина (підтримка 10/12-bit HDR)
    Bit_depth_luma_minus8      uint64  // 0=8-bit, 2=10-bit, 4=12-bit
    Bit_depth_chroma_minus8    uint64
    
    // Chroma format (4:2:0, 4:2:2, 4:4:4)
    Chroma_format_idc          uint64  // 1=4:2:0, 2=4:2:2, 3=4:4:4
}
```

### 🎯 Функція `GetH264Resolution` vs `GetH265Resolution`:

```go
// H.264: складна формула з макроблоками та кропом
func GetH264Resolution(sps []byte) (width uint32, height uint32) {
    widthInSample := (uint32(s.Pic_width_in_mbs_minus1) + 1) * 16
    widthCrop := uint32(s.Frame_crop_left_offset)*2 + uint32(s.Frame_crop_right_offset)*2
    width = widthInSample - widthCrop
    // ... аналогічно для висоти
}

// H.265: набагато простіше!
func GetH265Resolution(sps []byte) (width uint32, height uint32) {
    start, sc := FindStartCode(sps, 0)
    h265sps := H265RawSPS{}
    h265sps.Decode(sps[start+int(sc):])
    
    // Пряме читання у пікселях + врахування conformance window
    width = uint32(h265sps.Pic_width_in_luma_samples) - 
            uint32(h265sps.Conf_win_left_offset + h265sps.Conf_win_right_offset)
    height = uint32(h265sps.Pic_height_in_luma_samples) - 
             uint32(h265sps.Conf_win_top_offset + h265sps.Conf_win_bottom_offset)
    return
}
```

> 💡 **Перевага H.265**: немає множення на 16, немає `frame_mbs_only_flag` — менше місця для помилок у розрахунках.

## 🧩 4. `HEVCRecordConfiguration` (hvcc) — аналог `avcC` для H.265

Це **найскладніша частина**, яка генерує `HEVCDecoderConfigurationRecord` для fMP4 (ISO/IEC 14496-15, розділ 8.3.3.1.2):

### 📐 Структура hvcc (спрощено):

```
Offset  Size  Field
0       1     configurationVersion (always 1)
1       1     general_profile_space (2 bits) + general_tier_flag (1) + general_profile_idc (5)
2       4     general_profile_compatibility_flags (32 bits)
6       6     general_constraint_indicator_flags (48 bits)
12      1     general_level_idc
13      1     reserved (4 bits) + min_spatial_segmentation_idc (12 bits)
...     ...   chromaFormat, bitDepth, frameRate info
~23     1     numOfArrays (кількість типів NAL: VPS/SPS/PPS)
~24+    var   для кожного типу: array_completeness, NAL_unit_type, numNalus, [nalUnitLength + NALU]
```

### 🔧 Методи `UpdateSPS/UpdatePPS/UpdateVPS` — інкрементальне оновлення:

```go
func (hvcc *HEVCRecordConfiguration) UpdateSPS(sps []byte) {
    // 1. Знайти start code та розпарсити SPS
    start, sc := FindStartCode(sps, 0)
    var rawsps H265RawSPS
    rawsps.Decode(sps[start+int(sc):])
    
    // 2. Знайти існуючий масив для NAL type 33 (SPS)
    // 3. Перевірити, чи вже є такий SPS за ID та контентом
    // 4. Якщо ні — додати або оновити
    // 5. Оновити метадані: chromaFormat, bitDepth, PTL, VUI
}
```

### 🎯 Навіщо це у HLS:

| Сценарій | Без hvcc | З hvcc |
|----------|----------|--------|
| Ініціалізація fMP4 | FFmpeg не знає параметрів → помилка | `init.mp4` містить VPS/SPS/PPS → валідний сегмент |
| Зміна параметрів потоку | Нові SPS ігноруються → розсинхронізація | `UpdateSPS()` додає новий SPS у масив → плавний перехід |
| Мульти-бітрейт адаптація | Кожен варіант — окремий трек | Один hvcc з кількома SPS → ефективніше |

## ⚙️ 5. Складні підсистеми: що парситься, але не використовується

### 🔸 `Profile_tier_level` — профіль/рівень кодування:

```go
func Profile_tier_level(profilePresentFlag uint8, maxNumSubLayersMinus1 uint8, bs *BitStream) ProfileTierLevel {
    ptl.General_profile_idc = bs.Uint8(5)  // 0=Main, 1=Main10, 2=MainStill...
    ptl.General_level_idc = bs.Uint8(8)    // 30=720p30, 93=8K120...
    ptl.General_profile_compatibility_flag = bs.Uint32(32)  // бітова маска сумісності
    // ... ще 48 біт constraint flags
}
```

**Практичне значення**: Якщо `General_profile_idc == 1` (Main10) → відео 10-bit → ваш TTS-плейлист має сигналізувати клієнту про HDR підтримку.

### 🔸 `VUI_Parameters` — Video Usability Information:

```go
type VUI_Parameters struct {
    Aspect_ratio_info_present_flag uint8
    Vui_timing_info_present_flag   uint8
    Vui_num_units_in_tick          uint32  // для PTS розрахунку
    Vui_time_scale                 uint32  // clock frequency
    Min_spatial_segmentation_idc   uint64  // для parallel decoding
}
```

**Використання**: `Vui_time_scale / Vui_num_units_in_tick` → точний розрахунок тривалості сегменту для `#EXTINF` у HLS-плейлисті.

### 🔸 `skip_hrd_parameters` — HRD (Hypothetical Reference Decoder):

Це складна система управління буфером декодера. У вашому коді вона **пропускається** (`SkipBits`), тому що:

1. Для HLS не потрібна реалізація повного HRD
2. FFmpeg сам обробляє buffer management
3. Економія часу парсингу (критично для low-latency)

> 💡 **Порада**: Якщо ви реалізуєте власний декодер (наприклад, для прев'ю у WebSocket), доведеться реалізувати HRD для уникнення buffer underflow.

## 🎯 7. Інтеграція з вашим CCTV HLS Processor

### 📍 У `segmentAssembler` — уніфікована обробка H.264/H.265:

```go
func (sa *SegmentAssembler) processVideoNALU(codecID codec.CodecID, nalu []byte) {
    switch codecID {
    case codec.CODECID_VIDEO_H264:
        hdr := parseH264Header(nalu)
        if hdr.Nal_unit_type == codec.H264_NAL_SPS {
            sa.updateH264ParamSets(nalu)
        }
        
    case codec.CODECID_VIDEO_H265:
        hdr := parseH265Header(nalu)  // 2-байтовий заголовок
        switch hdr.Nal_unit_type {
        case codec.H265_NAL_VPS:
            sa.updateH265VPS(nalu)
        case codec.H265_NAL_SPS:
            sa.updateH265SPS(nalu)
        case codec.H265_NAL_PPS:
            sa.updateH265PPS(nalu)
        }
        
        // Детекція ключового кадру для сегментації
        if hdr.IsIRAP() && sa.shouldStartNewSegment() {
            sa.finalizeCurrentSegment()
            sa.startNewSegment()
        }
    }
}
```

### 📍 У `createTSSegment` — генерація hvcc для fMP4:

```go
func createH265InitSegment(vps, sps, pps [][]byte) ([]byte, error) {
    hvcc := codec.NewHEVCRecordConfiguration()
    
    for _, v := range vps { hvcc.UpdateVPS(v) }
    for _, s := range sps { hvcc.UpdateSPS(s) }
    for _, p := range pps { hvcc.UpdatePPS(p) }
    
    extradata, err := hvcc.Encode()
    if err != nil {
        return nil, fmt.Errorf("failed to encode hvcc: %w", err)
    }
    
    // Додати extradata у fMP4 'hvc1' track
    return buildFMP4InitSegment(extradata, "hvc1"), nil
}
```

```go
func calculateSegmentDurationH265(sps *codec.H265RawSPS, vui *codec.VUI_Parameters) time.Duration {
    if vui.Vui_timing_info_present_flag == 0 {
        return defaultSegmentDuration  // fallback

    clockFreq := float64(vui.Vui_time_scale) / float64(vui.Vui_num_units_in_tick)
    // Для 10-секундного сегменту:
    return time.Duration(float64(10 * clockFreq))
}
```

## 🧭 Висновок: чому H.265 складніший, але вартий того

| Аспект | H.264 | H.265 | Вигода для вашого проекту |
|--------|-------|-------|---------------------------|
| **Компресія** | Базова | +25-50% ефективність | Менший бітрейт → менше навантаження на мережу |
| **Роздільна здатність** | До 4K | До 8K+ | Підтримка майбутніх камер |
| **HDR підтримка** | Обмежена | Нативна 10/12-bit | Краща якість для нічного відеоспостереження |
| **Парсинг** | Простіший | Складніший (VPS, temporal layers) | Більше метаданих для адаптивного стрімінгу |
| **Сумісність** | Універсальна | Вимагає новіші декодери | Поступовий перехід: підтримувати обидва кодеки |

> 🔑 **Головна ідея**: Цей код — **інвестиція в майбутнє**. Поки більшість CCTV-камер використовують H.264, нові моделі вже переходять на H.265. Маючи цей парсер, ваш HLS Processor:
> 1. Може обробляти обидва кодеки через єдиний інтерфейс
> 2. Екстрагує більше метаданих для розумної сегментації
> 3. Готовий до HDR, 4K/8K, adaptive bitrate без переписування ядра

💡 **Фінальна порада**: Додайте інтеграційний тест, який порівнює вихід `GetH264Resolution` та `GetH265Resolution` для одного й того ж відео, закодованого в обох форматах — це гарантує, що ваші HLS-сегменти матимуть однакову роздільну здатність незалежно від кодеку.
